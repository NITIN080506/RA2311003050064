## Stage 1

Overview

This document captures the design and implementation notes for a Notifications microservice intended to deliver short messages to students (placements, events, results). It contains the API contract consumed by frontends, the persistent storage design, and performance guidance for real-time delivery and high-volume operations.

Stage 1 — API contract and real-time delivery

Summary

The Notifications service exposes a small set of REST endpoints for listing, reading, creating, and bulk-triggering notifications. It also offers a real-time push channel so logged-in clients receive new notifications immediately.

Security and headers

- All client requests must include `Authorization: Bearer <token>`.
- All JSON requests and responses use `Content-Type: application/json` and return standard HTTP status codes.

Core endpoints

- List notifications

  - Method: GET
  - Path: /notifications
  - Query params: `studentId` (required), `page` (default 1), `pageSize` (default 20), `onlyUnread` (optional)
  - Description: Returns a paginated list of notifications for the student. By default returns newest first.
  - Success (200):

    {
      "notifications": [
        {
          "id": "d146095a-0d86-4a34-9e69-3900a14576bc",
          "studentId": 1042,
          "type": "Result",
          "title": "Mid-sem",
          "message": "Your mid-sem marks are available",
          "isRead": false,
          "priority": 1,
          "metadata": null,
          "createdAt": "2026-04-22T17:51:30Z"
        }
      ],
      "page": 1,
      "pageSize": 20,
      "total": 123
    }

- Get a single notification

  - Method: GET
  - Path: /notifications/:id
  - Description: Reads a single notification object by id.
  - Success (200): returns the notification JSON.

- Mark read/unread

  - Method: PATCH
  - Path: /notifications/:id/read
  - Body: `{ "isRead": true }`
  - Description: Update read state; returns updated object (200).

- Create notification (internal/admin)

  - Method: POST
  - Path: /notifications
  - Body:

    {
      "studentId": 1042,
      "type": "Placement",
      "title": "CSX Corporation hiring",
      "message": "Visit campus drive portal to apply",
      "metadata": { "company": "CSX" },
      "priority": 3
    }

  - Description: Creates and persists a notification. Use this for server-generated or admin messages. Returns 201 with created id and createdAt.

- Notify all (bulk trigger)

  - Method: POST
  - Path: /notify_all
  - Body:

    {
      "studentIds": [1,2,3],
      "type": "Placement",
      "title": "All students: placements open",
      "message": "Apply via portal",
      "metadata": {}
    }

  - Description: Asynchronously enqueues a bulk job. The endpoint responds 202 Accepted and returns a job id for tracking.

Real-time delivery contract

- Transport: WebSocket (socket.io recommended) or Server-Sent Events for unidirectional push.
- Auth: client opens socket and sends an `authenticate` message containing bearer token; server validates and accepts the connection.
- Events:

  - `notification.created` — payload: the notification object (same schema as above). Emitted to clients subscribed to that student id.
  - `notification.updated` — payload: updated notification object (e.g., when marked read).

Example WebSocket flow

1. Client connects to `/ws` and emits `authenticate` with `{ token: "Bearer <token>" }`.
2. Server validates token and subscribes the socket to a channel `student:<studentId>`.
3. When a new notification exists for that student, server emits `notification.created` to `student:<studentId>`.

Errors and edge cases

- 401 for missing or invalid token.
- 400 for invalid request payloads.
- 429 if `notify_all` is abused; enforce rate limiting for bulk operations.

Stage 2 — Storage choice and schema (human explanation)

Decision summary

Choose PostgreSQL for the primary notification store. It gives predictable transactions, rich indexing, and `JSONB` for flexible metadata. For extremely high write volume, a NoSQL store (e.g., DynamoDB) is an option, but it requires more custom querying logic for the priority inbox.

Table design (explanation of each field)

- `id` (UUID): unique identifier generated by the database.
- `student_id` (BIGINT): the student the notification belongs to.
- `notification_type` (enum): one of `Placement`, `Event`, `Result` — used to weight priority.
- `title`, `message` (TEXT): contents of the notification.
- `is_read` (BOOLEAN): whether the student has read the notification.
- `priority` (SMALLINT): small integer from 0..n to carry explicit importance.
- `metadata` (JSONB): optional object for extensible fields.
- `created_at`, `updated_at` (TIMESTAMPTZ): timestamps to order and manage retention.

Why these indexes matter

- Lookups are primarily by `student_id` and often filter `is_read = false` and order by `created_at DESC`. A composite index `(student_id, created_at DESC)` supports ordered retrieval for a student. A partial index with `WHERE is_read = false` speeds unread-only queries without costing indexing space for read rows.

Retention and partitioning

- Keep recent notifications online and archive older ones if needed. Partitioning by range on `created_at` makes dropping/archiving old data efficient.

Migration SQL (example)

```
CREATE TYPE notification_type AS ENUM ('Placement','Event','Result');

CREATE TABLE notifications (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  student_id BIGINT NOT NULL,
  notification_type notification_type NOT NULL,
  title TEXT NOT NULL,
  message TEXT NOT NULL,
  is_read BOOLEAN NOT NULL DEFAULT false,
  priority SMALLINT NOT NULL DEFAULT 0,
  metadata JSONB,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE INDEX idx_notifications_student_createdat ON notifications (student_id, created_at DESC);
CREATE INDEX idx_notifications_student_unread_createdat ON notifications (student_id, created_at DESC) WHERE is_read = false;
```

Operational notes

- Avoid adding indexes for every column: each index slows writes and increases storage. Add indexes aligned with actual query patterns.
- Maintain a small Redis cache for per-student unread counts to support UI badges without COUNT queries.

Stage 3 — Performance analysis and targeted SQL changes (human explanation)

Context

Developers reported a slow query used to fetch unread notifications for a student. The problematic query is:

```
SELECT * FROM notifications
WHERE studentID = 1042 AND isRead = false
ORDER BY createdAt DESC;
```

Why it can be slow (plain language)

- If the database must scan many rows to find matching `studentID`, and then sort them by `createdAt`, the database does extra work: scanning, filtering, and sorting. Sorting large sets is CPU and memory intensive and can spill to disk.
- An index that only helps find rows (e.g., index on `student_id`) may still force a sort if `created_at` ordering is not covered. A combined index that includes `created_at` avoids the extra sort.

What to change and why

- Add a partial, composite index tailored to this exact pattern. The index below lets the database fetch rows in the correct order directly from the index, avoiding a separate sort step.

```
CREATE INDEX idx_notifications_student_unread_createdat ON notifications (student_id, created_at DESC) WHERE is_read = false;
```

- If the application often fetches both read and unread notifications ordered by time, add a full composite index without the partial predicate:

```
CREATE INDEX idx_notifications_student_createdat ON notifications (student_id, created_at DESC);
```

How to validate

- Run `EXPLAIN ANALYZE` on the query before and after adding the index. Before the index, you will likely see either a sequential scan or an index scan followed by a `Sort`. After the index, you should see an `Index Scan` that reads rows in `created_at` order and no large sort step.

Priority Inbox (Top-N) strategy

Goal

Return the top N unread notifications for a student where priority is a combination of the notification type weight and recency.

Simple runtime scoring

We can compute a score at query time combining type weight and a recency decay:

```
SELECT id, notification_type, title, message, is_read, priority, metadata, created_at,
  (
    CASE notification_type
      WHEN 'Placement' THEN 3
      WHEN 'Result' THEN 2
      WHEN 'Event' THEN 1
      ELSE 0
    END
    + (1.0 / (EXTRACT(EPOCH FROM (now() - created_at))/3600 + 1))
  ) AS score
FROM notifications
WHERE student_id = $1
  AND is_read = false
ORDER BY score DESC, created_at DESC
LIMIT 10;
```

Tradeoffs

- Computing score on every request is simple but prevents efficient use of B-tree indexes for ordering by score. It is acceptable for moderate traffic and small page sizes.
- For scale, maintain a `score` column computed at write time (and periodically updated) and index `(student_id, score DESC)` or use a Redis sorted set per student for real-time Top-N performance.

## Stage 2

Choice of persistent storage

- Recommendation: PostgreSQL (relational)

  - Rationale: ACID guarantees for notifications, powerful indexing (including partial and expression indexes), JSONB for flexible metadata, support for partitioning, and mature tooling for migrations and backups. Handles 5M rows comfortably with proper indexing and partitioning.

- Alternative: DynamoDB or another key-value store

  - Rationale: easier horizontal scaling and single-digit millisecond writes at very high scale. Tradeoffs: more complex queries (no joins), eventual consistency options, and additional engineering for complex reads (e.g., priority scoring across recency).

Schema (Postgres)

```
CREATE TYPE notification_type AS ENUM ('Placement','Event','Result');

CREATE TABLE notifications (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  student_id BIGINT NOT NULL,
  notification_type notification_type NOT NULL,
  title TEXT NOT NULL,
  message TEXT NOT NULL,
  is_read BOOLEAN NOT NULL DEFAULT false,
  priority SMALLINT NOT NULL DEFAULT 0,
  metadata JSONB,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE INDEX idx_notifications_student_createdat ON notifications (student_id, created_at DESC);
CREATE INDEX idx_notifications_student_unread_createdat ON notifications (student_id, created_at DESC) WHERE is_read = false;
CREATE INDEX idx_notifications_type_createdat ON notifications (notification_type, created_at DESC);
CREATE INDEX idx_notifications_createdat ON notifications (created_at DESC);
CREATE INDEX idx_notifications_priority_createdat ON notifications (priority DESC, created_at DESC);

-- Consider partitioning by range on created_at for very large tables:
-- CREATE TABLE notifications_y2026 PARTITION OF notifications FOR VALUES FROM ('2026-01-01') TO ('2027-01-01');
```

Key queries

- Insert a notification (application/server-side):

```
INSERT INTO notifications (student_id, notification_type, title, message, priority, metadata)
VALUES ($1, $2, $3, $4, $5, $6) RETURNING id, created_at;
```

- Fetch unread notifications for a student (paged):

```
SELECT id, notification_type, title, message, is_read, priority, metadata, created_at
FROM notifications
WHERE student_id = $1
  AND is_read = false
ORDER BY created_at DESC
LIMIT $2 OFFSET $3;
```

- Fetch all notifications for a student (paged):

```
SELECT id, notification_type, title, message, is_read, priority, metadata, created_at
FROM notifications
WHERE student_id = $1
ORDER BY created_at DESC
LIMIT $2 OFFSET $3;
```

- Find students who got a placement notification in the last 7 days:

```
SELECT DISTINCT student_id
FROM notifications
WHERE notification_type = 'Placement'
  AND created_at >= now() - interval '7 days';
```

Notes on scale and common problems

- The naive query `WHERE studentID = 1042 AND isRead = false ORDER BY createdAt DESC` will be slow without an index that supports both the filter and the ordering. A composite index `(student_id, created_at DESC)` plus a partial index for unread rows (`WHERE is_read = false`) addresses this.
- Adding indexes on every column is not safe: indexes slow down writes and increase storage. Add indexes only for frequent read patterns (e.g., student lookups, unread filters, type/time queries).
- For very large tables, use partitioning (time-range) and archive old partitions to cheaper storage. Combine with a TTL/retention policy for old notifications.
- Consider maintaining a separate `student_unread_counts` cache table or Redis counter to avoid COUNT(*) on notifications for unread badges.

Design choices

- Use `JSONB` for `metadata` to allow flexible fields without schema migrations.
- Use transactions when creating a notification and enqueuing downstream work (e.g., push to websocket queue) to avoid partial failures.
- For `notify_all` flows, write to DB first and then enqueue delivery jobs (see Stage 5) to ensure persistence even if delivery fails.

## Stage 3

Problem analysis

The slow query example:

Why this can be slow

- If there is no composite index that covers both the WHERE filters and the ORDER BY, Postgres must scan many rows and perform a sort. If the table has millions of rows, this becomes expensive.
- An index on only `student_id` helps find rows for the student, but does not satisfy ordering by `created_at DESC` efficiently unless the index is `(student_id, created_at DESC)`.
- Scanning the whole table or a large subset, then sorting, will use CPU and possibly spill to disk if the sort doesn't fit in memory.
- Adding an index on every column is not a solution: indexes increase write cost and storage. Add indexes targeted to query patterns.

Index recommendations

- Best index for this pattern (unread per student ordered by time):

CREATE INDEX idx_notifications_student_unread_createdat ON notifications (student_id, created_at DESC) WHERE is_read = false;

- Alternative (if you query both read/unread often):

CREATE INDEX idx_notifications_student_createdat ON notifications (student_id, created_at DESC);

- For queries that filter by `notification_type` and time, use:

CREATE INDEX idx_notifications_type_createdat ON notifications (notification_type, created_at DESC);

Optimized queries

- Fast unread fetch (uses partial index above):

SELECT id, notification_type, title, message, is_read, priority, metadata, created_at
FROM notifications
WHERE student_id = $1
  AND is_read = false
ORDER BY created_at DESC
LIMIT $2 OFFSET $3;

- If you need a stable limit for recent unread items, prefer `LIMIT` with a small `pageSize` to avoid large sorts.

EXPLAIN notes

- With the partial index, `EXPLAIN` should show an `Index Scan` on `idx_notifications_student_unread_createdat` and no large `Sort` step.

Priority Inbox (Top-N) strategy

Requirements: return top-N unread notifications ordered by a score combining type weight and recency.

Simple in-query scoring (works without precomputed scores):

SELECT id, notification_type, title, message, is_read, priority, metadata, created_at,
  (
    CASE notification_type
      WHEN 'Placement' THEN 3
      WHEN 'Result' THEN 2
      WHEN 'Event' THEN 1
      ELSE 0
    END
    + (1.0 / (EXTRACT(EPOCH FROM (now() - created_at))/3600 + 1))
  ) AS score
FROM notifications
WHERE student_id = $1
  AND is_read = false
ORDER BY score DESC, created_at DESC
LIMIT 10;

Notes on this score

- `CASE` encodes type weight (placement > result > event).
- The recency term `1/(hours_since+1)` gives a value in (0,1] so recent items get a modest boost.
- Tuning: adjust the type weights and denominator to change the recency decay.

Scaling and efficiency for Top-N

- Calculating score on every query prevents use of standard B-tree indexes for ordering. For large-scale systems maintain a precomputed `score` column updated on insert and periodically via a background job, or maintain a per-student Redis sorted set of notification IDs keyed by score.
- Option A (denormalized score column): add `score DOUBLE PRECISION` to `notifications`, compute on insert and update periodically; then index `(student_id, score DESC)` for fast Top-N.

ALTER TABLE notifications ADD COLUMN score DOUBLE PRECISION;
CREATE INDEX idx_notifications_student_score ON notifications (student_id, score DESC);

- Option B (Redis sorted sets): when a notification is created, push its computed score to `ZADD student:{id}:notifications {score} {notification_id}`. Top-N is `ZREVRANGE student:{id}:notifications 0 9 WITHSCORES`.

Tradeoffs

- Denormalized score: increased write cost and storage but fast SQL reads.
- Redis sorted sets: very fast Top-N reads, eventual consistency between DB and Redis, requires extra operational resources.

## Stage 4

Performance improvements and caching strategies

Goal

Keep read latency low for per-student notification views (especially unread counts and Top‑N lists) while maintaining durable writes and correctness for bulk operations.

Key strategies (high level)

- Short-term cache: use a per-student Redis cache for frequently requested small items (unread count, Top‑N ids). TTL of 10–60 seconds is reasonable to reduce DB pressure while keeping UIs near-real-time.
- Long-term store: PostgreSQL remains the source of truth for all notifications. Use partitioning and targeted indexes to keep queries fast.
- Denormalized counters: maintain a `student_unread_counts` table or Redis counter updated transactionally when writes occur to avoid COUNT(*) scans.
- Materialized Top‑N: maintain a denormalized `score` column updated at write time or by a background job to allow indexed Top‑N reads.

Detailed recommendations

- Redis for fast reads

  - Use Redis keys per student: `student:<id>:unread_count` (integer) and `student:<id>:top_notifications` (sorted set of notification IDs by score).
  - On notification create: increment `student:<id>:unread_count` and `ZADD` into the sorted set. On mark-read: decrement and `ZREM` as needed.
  - Set short TTLs on caches where eventual consistency is acceptable, and treat Redis as a read cache with Postgres as authoritative for reconciliation.

- Denormalized SQL counters

  - Create `student_unread_counts(student_id BIGINT PRIMARY KEY, unread_count BIGINT)` and update inside the same DB transaction that inserts notifications or marks them read. This guarantees consistency at the cost of a small extra write.

- Partitioning and retention

  - Partition `notifications` by range on `created_at` (monthly or quarterly depending on volume). This makes archival and deletion efficient.
  - Retention policy: move partitions older than N months to an archive schema or export to object storage.

- Index hygiene

  - Keep the partial index for unread queries: `(student_id, created_at DESC) WHERE is_read = false`.
  - Add an index on `(student_id, score DESC)` if you denormalize scores.
  - Regularly monitor index bloat and reindex/maintain VACUUM schedules.

- WebSocket / real-time scaling

  - Use socket.io with a Redis adapter to scale across multiple backend instances. Each socket joins a `student:<id>` room so broadcasts target only interested sockets.
  - Architecture: multiple app servers behind a load balancer → Redis pub/sub adapter → socket cluster. When a notification is created, the app publishes to Redis and all servers deliver to connected sockets.
  - Protect the real-time channel with authentication and rate limits to prevent abuse.

- Bulk operations and backpressure

  - `notify_all` should be an asynchronous job: write notifications to DB in batches, enqueue delivery tasks to a worker queue (e.g., Bull/Redis, RabbitMQ, or SQS), and return 202 with a job id.
  - Worker behavior: process in batches (e.g., 500–2000 writes per batch), back off on failures, and implement retries with exponential backoff and a poison queue for persistent failures.

- Observability and safe deployments

  - Metrics to collect: notifications written/sec, notifications delivered/sec, unread_count cache hits/misses, Top‑N latency P50/P95/P99, socket connection count, job queue backlog.
  - Alerts: queue backlog growth, high retry rates, increased Top‑N latency, Redis memory pressure.
  - Run schema migrations behind a migration tool and use feature flags for progressive rollout of denormalized score columns.

Tradeoffs and choices

- Redis sorted sets provide the fastest Top‑N reads but add operational complexity and eventual consistency between DB and Redis.
- Denormalized SQL score column increases write cost and storage but keeps the system simpler to operate if read traffic is moderate.
- Partial indexes reduce read cost for unread queries but make writes slightly slower; the tradeoff is worth it when read traffic dominates.

Practical next steps (short checklist)

1. Add `student_unread_counts` table and update application write logic to maintain it in DB transactions.
2. Add Redis keys and implement cache read-through with short TTLs for Top‑N and unread counts.
3. Implement reliable `notify_all` as an async job with batching, retries, and poison handling (Stage 5).

## Stage 5

Goal

Implement a reliable `notify_all` flow that persists notifications, performs batched writes, and retries failed delivery without losing messages.

Overview

The `notify_all` endpoint should not attempt to deliver 50k notifications inline. Instead it should create a persistent job record and return 202. A worker will pick up jobs, write notifications in batches, push delivery events, and handle retries with exponential backoff.

Job table (example migration)

CREATE TABLE notify_jobs (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  requester_id UUID NULL,
  payload JSONB NOT NULL,
  status TEXT NOT NULL DEFAULT 'pending',
  attempts INT NOT NULL DEFAULT 0,
  batch_size INT NOT NULL DEFAULT 500,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  started_at TIMESTAMPTZ,
  finished_at TIMESTAMPTZ
);

CREATE TABLE notify_job_errors (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  job_id UUID REFERENCES notify_jobs(id),
  error TEXT,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

Enqueue behavior (`/notify_all` handler)

- Validate requester and payload (studentIds array, message, type).
- Insert a row into `notify_jobs` with `payload` containing the studentIds and message.
- Return 202 with `{ jobId }`.

Worker behavior (high level)

1. Poll `notify_jobs` for `status = 'pending'` using `SELECT ... FOR UPDATE SKIP LOCKED` and mark `status = 'processing'` with `started_at`.
2. For the selected job, read `payload.studentIds` and process in chunks (e.g., 500 ids).
3. For each chunk:
   - Begin DB transaction.
   - Insert multiple `notifications` rows in one multi-row `INSERT` using `RETURNING id, student_id, created_at`.
   - Optionally push real-time events (publish to Redis or call internal push API) or call `Log` for observability.
   - Commit transaction.
4. If a chunk fails, roll back, increment `attempts` on the job, and retry with exponential backoff. After `MAX_ATTEMPTS` mark job `failed` and record error to `notify_job_errors`.
5. When all chunks succeed, set job `status = 'completed'` and `finished_at`.

Batching and efficiency

- Multi-row INSERT reduces DB round-trips.
- Use `COPY` for extremely high volumes if supported by the stack, or bulk-loading via prepared statements.
- Use server-side prepared statements to reuse query plans.

Retries and safety

- Use `attempts` counter and `updated_at` to implement exponential backoff. For transient failures increment `attempts` and schedule next attempt later.
- For permanent failures (e.g., malformed payload) mark `failed` immediately and capture error details.

Observability

- Emit metrics: jobs created, jobs processed, failed jobs, average processing time, notifications created per second.
- Log job progress and errors using the logging middleware.

Prototype files

- `migrations/JobsNotify.sql` — migration to create job tables.
- `src/NotifyAll.ts` — minimal TS handler that inserts a job row.
- `src/worker.ts` — minimal polling worker implementation that processes jobs in batches using Postgres transactions, with retry logic and logging.

## Stage 6

Goal

Implement a Priority Inbox endpoint that returns the top-N unread notifications for a student, ranked by a combination of notification type (Placement > Result > Event) and recency.

Strategy

Two approaches:

1. In-query scoring (simple, works for moderate traffic)
   - Compute score as: `typeWeight + recency bonus` at query time.
   - No extra column or index needed.
   - Acceptable latency for N < 100 and typical student notification counts (< 10,000 unread).

2. Denormalized score (for scale)
   - Add a `score` column to `notifications` table.
   - Compute and store score at write time or via a background batch job.
   - Index `(student_id, score DESC)` for fast Top-N reads.
   - Higher operational cost but enables consistent, indexed lookups.

Implementation approach (prototype)

Use in-query scoring for the prototype (simpler, fewer moving parts). If monitoring shows Top-N latency is high, migrate to denormalized scores using a migration + background job.

Top-N query (in-query scoring)

SELECT id, notification_type, title, message, is_read, priority, metadata, created_at,
  (
    CASE notification_type
      WHEN 'Placement' THEN 3
      WHEN 'Result' THEN 2
      WHEN 'Event' THEN 1
      ELSE 0
    END
    + (1.0 / (EXTRACT(EPOCH FROM (now() - created_at))/3600 + 1))
  ) AS score
FROM notifications
WHERE student_id = $1
  AND is_read = false
ORDER BY score DESC, created_at DESC
LIMIT $2;

Scoring explanation

- Type weight: 3 for Placement, 2 for Result, 1 for Event.
- Recency bonus: `1 / (hours + 1)` produces a value in (0, 1). A notification 1 hour old gets +0.5, 23 hours old gets ~+0.04. Tunes the importance of freshness.
- Combined: notifications are ranked first by type, then by how recent they are.
